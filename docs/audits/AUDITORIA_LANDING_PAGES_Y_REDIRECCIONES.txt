RESUMEN EJECUTIVO

La aplicacion no tiene hoy una landing publica institucional activa. Existe un componente PublicLanding con contenido generico de "Analisis de Calidad", pero no esta importado ni renderizado por App.jsx. En la practica, la ruta "/" para usuarios no autenticados cae en PublicApp y termina mostrando LoginForm, no PublicLanding.

La separacion entre "portal del colaborador" y "gestion interna" existe solo parcialmente despues de autenticar, dentro de MainApp, mediante allowedSections, sidebarSections y DynamicWelcomePortal. No hay un router declarativo ni guards de rutas dedicados. El frontend usa window.location.pathname, window.history.pushState y estado local para simular rutas.

La redireccion post-login no distingue roles de forma explicita. App.jsx fija siempre postLoginTarget = "panel". Despues MainApp fuerza /inicio para cualquier usuario autenticado. La experiencia por rol aparece porque /inicio renderiza DynamicWelcomePortal y ese componente muestra una vista distinta segun role. Por lo tanto, todos los roles terminan tecnicamente en la misma ruta inicial: /inicio.

La fuente de verdad mas fuerte del rol es public.profiles.role en Supabase, usada por useAuth, MainApp y los middlewares backend. Sin embargo, existen fallbacks a app_metadata.role y a "user"; ademas, el frontend usa "user" como rol base, mientras el requerimiento funcional menciona "usuario". En codigo real el rol comun es "user", no "usuario".

El backend protege endpoints criticos con authenticateToken, requireAdmin y requireAdminOrNutritionist. Para Documentos SGC, la ruta solo declara authenticateToken, pero los controllers resuelven rol y aplican canViewByRole/canManageByRole. Por lo tanto, hay proteccion backend real, aunque distribuida.

DIAGNOSTICO PRINCIPAL

Por que se sigue viendo una unica entrada generica:

1. PublicLanding existe pero no esta conectada.
   - frontend/src/components/PublicLanding.jsx:32-244 define la landing generica.
   - frontend/src/App.jsx no importa PublicLanding.
   - frontend/src/App.jsx:547-553 devuelve LoginForm como fallback de PublicApp.

2. "/" no renderiza landing institucional.
   - frontend/src/App.jsx:50 incluye "/" como public path.
   - frontend/src/App.jsx:484-553 define PublicApp.
   - No hay branch para currentPath === "/".
   - Resultado: "/" cae en el return final de PublicApp y muestra LoginForm.

3. La separacion por rol ocurre dentro de /inicio, no como recorridos separados.
   - frontend/src/App.jsx:560-563 setea postLoginTarget en "panel".
   - frontend/src/App.jsx:257-260 fuerza navigateToSection("panel") por usuario autenticado.
   - sectionPathMap.panel es "/inicio" en frontend/src/App.jsx:28-30.
   - frontend/src/App.jsx:406-413 renderiza DynamicWelcomePortal para currentSection "panel".
   - frontend/src/components/DynamicWelcomePortal.jsx:148-157 bifurca UI segun role.

4. No existe una funcion central reutilizable para decidir landing post-login por rol.
   - Hay normalizeRole en App.jsx:76-78.
   - Hay resolveAuthRedirectUrl en frontend/src/lib/authRedirect.js:18-47, pero solo decide URL/origin de Supabase, no destino interno por rol.
   - No existe getPostLoginRoute(role) ni equivalente.

AUTENTICACION Y CARGA DEL PERFIL

Flujo paso a paso:

1. App monta desde main.jsx.
2. useAuth inicia loading=true (useAuth.js:20-23).
3. bootstrap llama supabase.auth.getSession (useAuth.js:74-75).
4. Si hay session.user, llama ensureProfileFromAuthUser (useAuth.js:77).
5. ensureProfileFromAuthUser consulta public.profiles por id (useAuth.js:34-38).
6. Si hay error, devuelve null (useAuth.js:40-42).
7. Si no hay profile, inserta profiles con role "user" e is_active true (useAuth.js:44-57).
8. Si existe profile sin full_name y metadata trae nombre, lo actualiza (useAuth.js:60-69).
9. mapSupabaseUser arma user con role profile.role || appMetadata.role || "user" (useAuth.js:12-17).
10. Guarda access_token en localStorage o lo borra (useAuth.js:82-87).
11. App deja de mostrar "Cargando..." (App.jsx:565-574).
12. Si no hay user o hay recovery/reset, renderiza PublicApp (App.jsx:579-580).
13. Si hay user, renderiza MainApp (App.jsx:581-588).
14. MainApp vuelve a consultar profiles con id,email,full_name,role,is_active (App.jsx:176-191).
15. Si no hay error y data existe, usa ese profile (App.jsx:193-195).
16. Si falla o no existe, crea fallback local con role user.role || "user" e is_active true (App.jsx:196-203).
17. Mientras loadingProfile, muestra pantalla completa "Cargando perfil..." (App.jsx:465-470).
18. Al terminar, calcula role efectivo y allowedSections, y ejecuta redireccion/guard (App.jsx:88-158, 214-233, 253-260).

Tabla/endpoints de rol:
- Frontend lee directamente Supabase table public.profiles.
- Backend lee public.profiles en authenticateToken/requireAdmin/requireAdminOrNutritionist.
- SQL crea profiles por trigger desde auth.users.

Cuando se conoce role:
- Primer role conocido: al terminar useAuth bootstrap/onAuthStateChange.
- Role definitivo para UI privada: al terminar carga de profile en MainApp.
- Hay un estado intermedio potencial donde LoginForm mapSupabaseUser devuelve app_metadata.role || "user" antes de que useAuth/MainApp corrija.

Mientras profile carga:
- App global: "Cargando..." durante useAuth.loading.
- MainApp: pantalla completa "Cargando perfil..." durante loadingProfile.

Si profile no existe:
- useAuth intenta crearlo con role "user".
- Backend authenticateToken, en cambio, rechaza con 403 "Perfil de usuario no encontrado" si no existe.
- MainApp, si su consulta no trae data, usa fallback local role user.role || "user".
- Riesgo: frontend puede dejar entrar visualmente como user aunque backend luego rechace endpoints si el perfil realmente falta.

Si sesion existe pero profile falla:
- useAuth devuelve user desde app_metadata o "user".
- MainApp fallback local.
- Backend falla cerrado en error real de profiles: authenticateToken 500 o requireAdmin 500.

Si role viene vacio/mal escrito/desconocido:
- normalizeRole en frontend trim/lowercase.
- isAdmin solo "admin"; isNutritionist solo "nutricionista".
- Rol desconocido queda como no-admin y no-nutricionista, con experiencia de usuario comun.
- Backend requireAdmin/requireAdminOrNutritionist rechaza roles desconocidos.
- DB constraint nueva permite solo user/admin/nutricionista si fue aplicada.

Flicker visual:
- Hay loadingProfile de pantalla completa, por lo que se evita render de admin/user incorrecto durante la segunda carga.
- Posible transitorio menor: handleLoginSuccess setea user con role de app_metadata/"user"; MainApp muestra loadingProfile antes de renderizar secciones, mitigando flicker.

Estados de error:
- useAuth no expone error de profile.
- MainApp no muestra error de profile; usa fallback local.
- Backend si devuelve errores.
- Login/reset/forgot muestran errores propios.

SIMULACIONES

A. Usuario no autenticado
- App loading -> useAuth no encuentra sesion -> user null.
- App renderiza PublicApp.
- Si URL es "/" cae a LoginForm.
- Si intenta /cargar directo, normalizePublicPath devuelve /login en estado, pero no hay replaceState obligatorio.

B. Usuario comun autenticado
- useAuth obtiene/crea profile role "user".
- MainApp carga profile.
- effectiveRole user, isAdmin false, isNutritionist false.
- allowedSections: panel, declaration, policies.
- Redirige a /inicio.
- /inicio muestra UserView.
- Si pega /cargar, se redirige a /declaracion-salud.

C. Nutricionista autenticada
- role nutricionista.
- allowedSections: panel, declaration, policies, nutritionModules, certifications.
- Redirige a /inicio.
- /inicio muestra AdminView restringido: SGC y Certificaciones, sin Excel/Dashboard/Salud Admin/Usuarios.
- Si pega /cargar, se redirige a /declaracion-salud.

D. Admin autenticado
- role admin.
- allowedSections incluye panel, upload, charts, profile, tutorial, rules, adminUsers, adminHealthDeclarations, history, declarationHistory, nutritionModules, certifications.
- Redirige a /inicio.
- Puede navegar a panel interno completo.

E. Sesion valida pero perfil inexistente
- useAuth intenta insertar profile "user".
- Si insercion funciona: queda usuario comun.
- Si falla/sin data: frontend puede crear user fallback "user"; MainApp tambien fallback local.
- Backend authenticateToken rechazara endpoints si profiles sigue sin existir.

F. Rol desconocido
- Frontend: role desconocido se normaliza, no coincide con admin/nutricionista, queda experiencia usuario comun.
- Backend: requireAdmin y requireAdminOrNutritionist rechazan.
- DB constraint deberia impedirlo si aplicada.

ALCANCE MINIMO RECOMENDADO PARA DIVIDIR SIN DUPLICAR LOGICA

1. Definir constantes de roles y rutas iniciales:
- user -> /portal o /declaracion-salud o /inicio-colaborador.
- nutricionista -> /modulos-nutricionales o /inicio-gestion con subset.
- admin -> /inicio o /dashboard.

2. Crear una funcion unica getInitialPathForRole(role), usando public.profiles.role como fuente.

3. Conectar una landing publica real en "/".
- Puede reutilizar PublicLanding, pero su contenido deberia dejar de estar centrado solo en Excel si sera entrada institucional.
- Login deberia quedar en "/login".

4. Separar semanticamente home autenticado:
- Mantener componentes compartidos, pero evitar que nutricionista sea "AdminView" por nombre.
- Un PortalColaborador y un GestionInternaHome pueden reutilizar PortalCard.

5. Reemplazar o encapsular routing manual:
- Minimo: centralizar sectionPathMap, allowedSections y getSectionFromPath.
- Ideal: React Router con ProtectedRoute y RoleRoute.

6. Preservar returnTo para accesos directos:
- Si no autenticado entra a /modulos-nutricionales, despues del login podria volver si su rol lo permite.

7. Alinear "usuario" vs "user":
- Decidir valor persistido. Hoy debe ser "user".
- Si se quiere "usuario", requiere migracion DB, backend, frontend y policies.

RIESGOS PRIORIZADOS

ALTO
- "/" no muestra landing institucional; muestra login. Esto contradice el objetivo de entrada unica institucional.
- Post-login no distingue roles por ruta; todos van a /inicio.
- No existe funcion central de destino por rol; la logica esta repartida entre App.jsx y DynamicWelcomePortal.

MEDIO
- PublicLanding y SmartHome son codigo huerfano/prototipo y pueden generar confusion.
- Frontend usa fallbacks silenciosos a "user" si falla profiles, ocultando errores de permisos hasta llamadas API.
- "usuario" en requerimiento no coincide con "user" en codigo/DB.
- Backend no aplica trim en todos los middlewares de rol.

BAJO
- Rutas desconocidas publicas/privadas tienen fallbacks poco explicitos.
- /mi-declaraciones suena a usuario comun pero esta permitido solo para admin.
- LoginForm combina login y registro en el mismo componente, funcional pero acopla experiencias.

CONCLUSION

El estado actual no implementa una separacion clara entre landing institucional, portal del colaborador y gestion interna. Implementa una unica app autenticada con un home comun (/inicio) que cambia contenido segun rol. La landing publica generica existe pero esta desconectada, y la pantalla visible para "/" sin sesion es LoginForm.

La seguridad backend esta razonablemente cubierta para los modulos criticos, pero la experiencia y redireccion por rol estan incompletas desde el punto de vista funcional. El cambio minimo futuro no deberia duplicar logica: deberia centralizar roles/rutas iniciales, conectar una landing publica real, y convertir el home autenticado en destinos semanticamente separados por rol manteniendo componentes compartidos.
