1. RESUMEN EJECUTIVO
--------------------

El modulo "Documentos SGC" es una biblioteca documental interna para gestionar documentos del Sistema de Gestion de Calidad / Inocuidad.

Internamente mantiene nombres historicos de "nutrition-modules", pero funcionalmente corresponde a "Documentos SGC".

El modulo actualmente permite:

- Navegar una biblioteca jerarquica por carpetas y subcarpetas.
- Buscar carpetas y documentos.
- Filtrar documentos por apartado: Todos, Estrategias, Procedimientos, Registros.
- Crear carpetas.
- Renombrar carpetas.
- Mover carpetas.
- Archivar carpetas vacias.
- Crear documentos manualmente.
- Editar documentos.
- Mover documentos entre carpetas.
- Eliminar documentos.
- Adjuntar multiples archivos a documentos.
- Ver adjuntos.
- Descargar adjuntos.
- Eliminar adjuntos.
- Descargar el contenido de un documento como .txt.
- Exportar un documento a Excel.
- Importar una biblioteca documental completa desde un ZIP.
- Procesar notificaciones internas por email cuando se crean documentos.

El acceso esta dividido por rol:

- Admin: gestion completa.
- Nutricionista: lectura, navegacion y descarga.
- Otros roles: sin acceso funcional al modulo.

3. TABLAS PRINCIPALES
---------------------

3.1. nutrition_modules

Es la tabla principal de documentos SGC.

Campos relevantes usados por el backend:

- id
- title
- description
- content
- status
- module_type
- folder_id
- created_by
- created_at
- updated_at
- published_at

Observaciones:

- El nombre tecnico es "nutrition_modules", aunque representa documentos SGC.
- status se fuerza a "aprobado" en creacion y edicion.
- module_type solo admite:
  procedimiento
  registro
  estrategias
- folder_id es nullable.
- Los documentos antiguos sin folder_id siguen existiendo y aparecen en raiz / sin carpeta.

3.2. sgc_document_folders

Tabla de carpetas jerarquicas del modulo.

Campos:

- id
- name
- parent_id
- description
- status
- created_by
- created_at
- updated_at

Reglas:

- parent_id nullable indica carpeta raiz.
- parent_id apunta a otra carpeta.
- status puede ser:
  activo
  archivado
- No se permite parent_id igual al propio id.
- Hay trigger para evitar ciclos.
- Hay trigger para impedir archivar carpetas con subcarpetas activas o documentos.

3.3. nutrition_module_files

Tabla de adjuntos de documentos.

Campos:

- id
- module_id
- file_name
- file_path
- file_type
- file_size
- uploaded_by
- created_at

Reglas:

- module_id referencia nutrition_modules.
- file_path apunta al archivo en Supabase Storage.
- Los adjuntos se descargan desde backend, no directamente desde frontend.

3.4. document_email_notifications

Tabla de cola de notificaciones por email.

Uso:

- Al crear documentos manualmente se encola una notificacion.
- Un worker interno procesa notificaciones pendientes.
- La importacion ZIP crea documentos directamente en DB; por el codigo actual puede disparar triggers de DB si estan activos, pero no llama manualmente al worker post-create por cada archivo importado.

6. ENDPOINTS DEL MODULO
-----------------------

Todos los endpoints de usuario usan authenticateToken, salvo el endpoint interno de notificaciones.

6.1. Documentos

GET /api/nutrition-modules

- Lista documentos.
- Admin ve todos.
- No admin con permiso ve solo status = aprobado.
- Agrega filesCount contando adjuntos por documento.

GET /api/nutrition-modules/:id

- Obtiene un documento por id.
- Nutricionista solo puede ver si status = aprobado.

POST /api/nutrition-modules

- Crea documento.
- Solo admin.
- Campos requeridos:
  title
  moduleType / module_type
- Campos opcionales:
  description
  content
  folderId / folder_id
- status siempre queda "aprobado".
- published_at se setea en creacion.
- Valida que folder_id exista y no este archivado.
- Encola notificacion y dispara worker post-create en segundo plano.

PUT /api/nutrition-modules/:id

- Edita documento.
- Solo admin.
- Requiere title y moduleType.
- Puede actualizar folder_id si se envia.
- Mantiene published_at si ya existia.
- Fuerza status = aprobado.

PUT /api/nutrition-modules/:id/move

- Mueve documento a otra carpeta o a raiz.
- Solo admin.
- Valida que la carpeta destino exista y este activa.
- Si folderId es null, mueve a raiz.

DELETE /api/nutrition-modules/:id

- Elimina documento.
- Solo admin.
- Hace delete fisico en nutrition_modules.
- La tabla de archivos tiene referencia a modulo; segun migracion original nutrition_module_files usa on delete cascade.
- No elimina manualmente archivos de Storage en este controlador.

GET /api/nutrition-modules/:id/export/excel

- Exporta un documento a Excel.
- Admin y nutricionista.
- Nutricionista solo si el documento esta aprobado.
- Genera un XLSX con:
  Titulo
  Descripcion
  Contenido
  Fecha de creacion
  Fecha de actualizacion
  Fecha de aprobacion

GET /api/nutrition-modules/:id/download

- Descarga el contenido del documento como .txt.
- Admin y nutricionista.
- Nutricionista solo si aprobado.
- El archivo incluye:
  Titulo
  Apartado
  Descripcion
  Aprobado
  Contenido

6.2. Carpetas

GET /api/nutrition-modules/folders

- Lista carpetas.
- Admin y nutricionista.
- Por defecto excluye status archivado.
- Si query includeArchived=true, incluye archivadas.

POST /api/nutrition-modules/folders

- Crea carpeta.
- Solo admin.
- Requiere name.
- parentId es opcional.
- Si parentId existe, valida que la carpeta destino exista y este activa.

PUT /api/nutrition-modules/folders/:id

- Edita carpeta.
- Solo admin.
- Permite cambiar:
  name
  description
  parentId
  status
- Valida que name no este vacio.
- Valida status activo/archivado.
- Si se archiva, primero valida que la carpeta este vacia.
- Si se mueve, evita ciclos:
  no mover dentro de si misma
  no mover dentro de una descendiente

DELETE /api/nutrition-modules/folders/:id

- Archiva carpeta.
- Solo admin.
- No elimina fisicamente.
- Antes de archivar valida:
  sin subcarpetas activas
  sin documentos

6.3. Adjuntos

GET /api/nutrition-modules/:id/files

- Lista adjuntos de un documento.
- Admin y nutricionista si tienen acceso al documento.
- Devuelve:
  id
  moduleId
  fileName
  filePath
  fileType
  fileSize
  uploadedBy
  createdAt

POST /api/nutrition-modules/:id/files

- Sube adjuntos manuales.
- Solo admin.
- Usa multer en memoria.
- Permite hasta 10 archivos por request.
- Limite por archivo: 25 MB.
- Valida extensiones y MIME type.
- Valida duplicados dentro del mismo lote por nombre + tamaño.
- Valida duplicados contra adjuntos existentes del mismo documento por nombre + tamaño.
- Crea bucket si no existe.
- Sube cada archivo a Supabase Storage.
- Crea metadata en nutrition_module_files.
- Si falla guardar metadata, borra el archivo recien subido en Storage.

DELETE /api/nutrition-modules/files/:fileId

- Borra un adjunto.
- Solo admin.
- Busca metadata.
- Borra archivo en Storage.
- Luego borra metadata en DB.

GET /api/nutrition-modules/files/:fileId/download

- Descarga adjunto.
- Admin y nutricionista si tienen acceso al documento asociado.
- Descarga desde Supabase Storage.
- Devuelve Content-Type del file_type guardado.
- Devuelve Content-Disposition attachment.

6.4. Importacion ZIP

POST /api/nutrition-modules/import/zip/analyze

- Analiza ZIP.
- Solo admin.
- Recibe un unico archivo field "zip".
- Limite de ZIP comprimido: 100 MB.
- No escribe en DB ni Storage.
- Guarda temporalmente el ZIP y un manifest con token.
- Devuelve token y vista previa.

POST /api/nutrition-modules/import/zip/confirm

- Confirma importacion ZIP.
- Solo admin.
- Recibe token de analisis.
- Valida que el token exista, no haya expirado y pertenezca al mismo usuario.
- Reanaliza el ZIP antes de importar.
- Crea carpetas faltantes.
- Crea documentos.
- Sube adjuntos.
- Limpia temporales al finalizar.

6.5. Notificaciones

POST /api/internal/nutrition-modules/process-notifications

- Endpoint interno.
- No usa authenticateToken normal.
- Usa isWorkerAuthorized.
- Procesa cola document_email_notifications.
- batchSize minimo 1, maximo 100.

12.2. Analisis previo

Endpoint:

- POST /api/nutrition-modules/import/zip/analyze

No escribe en DB ni Storage.

Proceso:

1. Valida token y rol admin.
2. Recibe ZIP con multer en disco temporal.
3. Rechaza si no es .zip.
4. Abre ZIP con yauzl usando lazyEntries.
5. Recorre entradas.
6. Valida limites.
7. Normaliza separadores \ a /.
8. Rechaza rutas sospechosas.
9. Omite entradas de sistema.
10. Detecta carpeta envolvente DOCUMENTOS-SERVIFOOD y la quita.
11. Clasifica carpetas y archivos.
12. Valida archivos.
13. Calcula carpetas nuevas/existentes.
14. Calcula documentos importables.
15. Detecta duplicados.
16. Guarda manifest temporal con token.
17. Devuelve resumen.

Resumen devuelto:

- foldersNew
- foldersExisting
- documentsNew
- filesSkipped
- duplicates
- invalidFiles
- totalUncompressedBytes
- warnings
- limits
- files importables
- folderPaths
- token
- fileName

12.3. Validaciones de ZIP

Limites:

- ZIP comprimido: 100 MB.
- Total descomprimido: 500 MB.
- Archivo individual: 25 MB.
- Entradas maximas: 2500.
- Profundidad maxima: 25 segmentos.
- Token temporal: 60 minutos.

Entradas rechazadas u omitidas:

- Rutas vacias.
- Rutas absolutas.
- Rutas tipo C:/.
- Rutas con ../.
- __MACOSX.
- .DS_Store.
- Entradas que empiezan con ._.
- Archivos que empiezan con ~$.
- Archivos .tmp.
- Archivos .zip internos.
- Archivos sin extension permitida.
- Archivos vacios.
- Archivos mayores a 25 MB.
- Rutas con profundidad mayor a 25.
- ZIP con mas de 2500 entradas.
- ZIP con total descomprimido mayor a 500 MB.

12.4. Confirmacion de importacion

Endpoint:

- POST /api/nutrition-modules/import/zip/confirm

Proceso:

1. Valida token y rol admin.
2. Carga manifest temporal.
3. Valida que el manifest sea del mismo usuario.
4. Valida que no haya expirado.
5. Reanaliza el ZIP antes de importar.
6. Carga carpetas activas existentes.
7. Carga claves de adjuntos existentes.
8. Asegura bucket nutrition-modules.
9. Recorre archivos importables.
10. Para cada archivo:
    - crea o reutiliza carpeta destino
    - verifica duplicado en carpeta por nombre normalizado + tamaño
    - crea documento
    - lee archivo del ZIP
    - calcula hash SHA-256 parcial para ruta segura
    - sube archivo a Storage
    - crea metadata en nutrition_module_files
    - registra resultado
11. Si falla un archivo:
    - registra error
    - borra Storage si ya habia subido
    - borra documento si ya habia sido creado
    - continua con el siguiente
12. Al final borra ZIP temporal y manifest.

12.5. Campos por defecto en documentos importados

Cuando se importa un archivo desde ZIP:

- title:
  nombre del archivo sin extension.
- description:
  "Importado desde ZIP: <ruta interna>"
- content:
  string vacio.
- status:
  aprobado.
- module_type:
  registro.
- folder_id:
  id de la carpeta de la ruta.
- created_by:
  usuario admin que confirma.
- published_at:
  fecha actual.

Razon de module_type = registro:

- El esquema solo admite procedimiento, registro y estrategias.
- Desde un archivo individual no se puede inferir con certeza el apartado documental.
- "registro" es el valor conservador usado para no romper filtros ni constraints.

12.6. Deteccion de duplicados en ZIP

Dentro del ZIP:

- Compara carpeta normalizada + nombre normalizado + tamaño.
- Si coincide, omite como duplicado.

Contra datos existentes:

- Carga documentos y adjuntos existentes.
- Calcula carpeta destino + nombre normalizado + tamaño.
- Si ya existe, omite.

Notas:

- No sobrescribe archivos.
- No reemplaza documentos existentes.
- No borra documentos anteriores.
- No guarda hash persistente en DB.
- Usa hash SHA-256 solo para generar ruta de Storage mas unica.

12.7. Reporte final

El frontend muestra:

- Importados.
- Omitidos.
- Duplicados.
- Fallidos.
- Detalle de filas.

Permite descargar reporte .txt.

Contenido del reporte:

- Totales.
- Detalle por archivo:
  status
  path
  mensaje o documentId
- Advertencias.


13. STORAGE
-----------

Bucket:

- nutrition-modules

Creacion:

- ensureStorageBucketExists lista buckets.
- Si no existe, lo crea privado.

Rutas manuales:

- <documentId>/<Date.now()>-<safeName>

Rutas ZIP:

- <documentId>/<Date.now()>-<hash16>-<safeName>

Sanitizacion:

- Quita acentos.
- Reemplaza caracteres no alfanumericos, guion bajo o guion por _.
- Recorta base a 80 caracteres.
- Mantiene extension.

Descarga:

- Siempre pasa por backend.
- Backend usa file_path guardado.
- Devuelve archivo como attachment.

23. RIESGOS Y OBSERVACIONES
---------------------------

1. Eliminacion de documentos:

El controlador borra el documento de DB, pero no borra explicitamente los archivos fisicos de Storage asociados. Si la DB borra metadata por cascade, los objetos del bucket pueden quedar huerfanos.

2. Importacion ZIP y timeouts:

El procesamiento ZIP ocurre dentro de una request HTTP. Para ZIPs muy grandes o muchos archivos, podria chocar con timeouts de Render/proxy. Los limites actuales reducen el riesgo, pero una importacion real grande podria requerir worker en segundo plano.

3. MIME application/octet-stream:

Se admite para adjuntos porque algunos navegadores/SO reportan archivos Office o .msg como octet-stream. La extension sigue siendo validada.

4. Duplicados:

La deduplicacion persistente se basa en carpeta + nombre normalizado + tamaño. No usa hash persistente porque la tabla no tiene campo hash.

5. Notificaciones masivas:

Si hay triggers DB para document_email_notifications sobre nutrition_modules, una importacion masiva podria generar muchas notificaciones.

6. Descripciones de importacion:

Los documentos importados usan descripcion tecnica "Importado desde ZIP: ruta". No se infiere metadata documental.

7. Apartado de importacion:

Los documentos importados quedan como module_type = registro. Es conservador, pero puede requerir reclasificacion manual si el negocio necesita diferenciar procedimientos/estrategias.

25. CONCLUSION
--------------

El modulo Documentos SGC ya funciona como biblioteca documental jerarquica con carpetas, subcarpetas, documentos, adjuntos, descargas, exportacion Excel e importacion ZIP.

La seguridad principal esta en backend:

- validacion de token;
- resolucion de rol desde profiles;
- admin requerido para escritura;
- nutricionista limitado a lectura y descarga;
- validaciones de carpeta, archivo y ZIP;
- service role solo en backend.

La interfaz esta orientada a un explorador simple:

- breadcrumbs;
- boton volver;
- busqueda global;
- filtros por apartado;
- acciones por iconos;
- modales para crear/editar/mover/importar;
- estados de carga, error y exito.

Las areas a vigilar antes de uso masivo son:

- limpieza de objetos Storage al borrar documentos;
- posible volumen de notificaciones al importar cientos de archivos;
- timeouts si el ZIP real es muy pesado;
- reclasificacion posterior de documentos importados, ya que se crean como "registro".
