1. Causa exacta de la recarga anterior
- El reordenamiento no ejecutaba window.location.reload ni navegacion forzada.
- La lentitud y el parpadeo venian de frontend/src/components/NutritionModulesPage.jsx.
- La funcion persistOrder hacia await loadLibrary() despues de cada reorder exitoso.
- Ese loadLibrary volvia a solicitar carpetas y documentos completos tras cada movimiento.
- Ademas, canReorderCurrentView dependia de !savingOrder, por lo que bloqueaba movimientos consecutivos mientras persistia una request.

2. Archivos modificados
- frontend/src/components/NutritionModulesPage.jsx

3. Implementacion de actualizacion optimista
- Al mover con flechas o drag and drop se calcula el nuevo orden local.
- Se actualiza inmediatamente sortOrder en rows o folders.
- La tabla se reordena sin esperar al backend.
- No se ejecuta loadLibrary despues de reorder exitoso.
- No se recarga la pagina automaticamente.
- No se altera carpeta actual, breadcrumbs, filtros ni scroll por refetch completo.

4. Implementacion del debounce
- Se agrego debounce de 400 ms mediante ORDER_SAVE_DEBOUNCE_MS.
- Cada movimiento reinicia el timer.
- Varios movimientos rapidos dentro del mismo tipo y nivel consolidan una unica persistencia final cuando es posible.
- La persistencia usa el endpoint batch existente mediante reorderNutritionModuleItems.

5. Manejo de requests concurrentes
- Se agrego una cola local por clave type + parentFolderId.
- La cola separa orderedIds visibles, confirmedIds e inFlight.
- Si hay una request en curso y ocurre otro movimiento, se conserva el orden local mas reciente.
- Cuando responde la request anterior, si detecta que el orden local cambio, marca una nueva persistencia pendiente.
- Una respuesta vieja no pisa el orden visible mas nuevo.
- No se bloquea toda la tabla mientras se guarda.

6. Manejo de rollback
- Antes de persistir se conserva el ultimo orden confirmado por backend.
- Si falla la persistencia, se restaura visualmente ese orden confirmado.
- Se cancela cualquier debounce pendiente.
- Se limpia la cola de reorder.
- Se deshabilitan los controles de reordenamiento hasta recargar la pagina.
- No se deja visible un orden falso.

7. Mensaje user friendly implementado
- Se agrego un mensaje warning no tecnico:
  Titulo: No pudimos guardar el nuevo orden
  Texto: Restauramos la organización anterior para evitar inconsistencias. Recargá la página y volvé a intentarlo.
  Boton: Recargar página
- La pagina solo se recarga si el usuario pulsa el boton.
- No se muestra stack trace, JSON ni error tecnico crudo para el fallo de reorder.
