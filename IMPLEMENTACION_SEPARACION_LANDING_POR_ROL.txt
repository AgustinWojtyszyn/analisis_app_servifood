IMPLEMENTACION_SEPARACION_LANDING_POR_ROL
Fecha: 2026-06-10

RESUMEN DE CAMBIOS

Se implemento la separacion de experiencia publica y homes autenticados por rol conservando el ruteo manual existente basado en window.location.pathname, window.history.pushState, replaceState, popstate y estado local.

La ruta "/" sin sesion ahora muestra una landing institucional publica. El login queda en "/login" y el registro en "/register". Los usuarios autenticados ya no terminan todos en "/inicio": el rol real cargado desde public.profiles.role define el destino inicial.

DESTINOS POR ROL

user:
- Ruta inicial: /portal-colaborador
- Seccion inicial: collaboratorPortal
- Fallback: /portal-colaborador
- Accesos: inicio, declaracion de salud, politicas de seguridad

nutricionista:
- Ruta inicial: /gestion-interna
- Seccion inicial: internalManagement
- Fallback: /gestion-interna
- Accesos: inicio, declaracion de salud, politicas de seguridad, Documentos SGC, certificaciones

admin:
- Ruta inicial: /gestion-interna
- Seccion inicial: internalManagement
- Fallback: /gestion-interna
- Accesos: inicio, cargar archivos, historial, analisis detalle, graficos, perfil, tutorial, reglas, gestion de usuarios, declaracion de salud, politicas, declaraciones administrativas, Documentos SGC, certificaciones

Rol vacio, invalido o desconocido:
- Se normaliza de forma segura a user.
- No se cambio el valor tecnico persistido del rol comun: sigue siendo user.

DECISIONES TECNICAS

- No se agrego React Router.
- No se modifico backend.
- No se modifico SQL, migraciones, Supabase ni politicas RLS.
- No se cambiaron endpoints.
- No se uso email ni metadata como fuente definitiva de permisos.
- La decision final de destino autenticado ocurre despues de cargar el perfil en MainApp.
- Si la consulta de profile falla en MainApp, el fallback de UI es seguro y equivalente a user.
- Se uso replaceState para redirecciones automaticas y alias, para evitar historial inconsistente.
- Se uso pushState para navegacion iniciada por el usuario desde landing, sidebar o tarjetas.
- Se restauro frontend/dist/index.html despues del build para no dejar artefactos generados dentro del cambio fuente.
